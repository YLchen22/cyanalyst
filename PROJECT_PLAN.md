## 一、项目概述

构建一个**自动化投研工作流**，核心链路：


## 三、技术选型详解

### 3.1 资料库网站

**推荐方案：VitePress**

| 维度 | VitePress | Next.js | Docusaurus |
|------|-----------|---------|------------|
| 构建速度 | ⚡ 极快 | 中等 | 较慢 |
| Markdown 原生支持 | ⭐ 最佳 | 需配置 | ⭐ 优秀 |
| 搜索 | 内置全文搜索 | 需 Algolia | 内置/ Algolia |
| 学习成本 | 低 | 中高 | 中 |
| 部署体积 | 极小 | 中等 | 较大 |
| 适合场景 | **文档站点** | 复杂 Web 应用 | 文档站点 |

**结论**：VitePress 最适合「资料库 + Blog」定位，开箱即用的 Markdown 支持和全文搜索。

**托管方案：Cloudflare Pages（推荐）**

| 平台 | 免费额度 | 国内访问 | 构建限制 | 自定义域名 |
|------|----------|---------|----------|------------|
| **Cloudflare Pages** | 无限带宽，500次构建/月 | ⭐⭐⭐ 较好 | 20min/次 | ✅ |
| Vercel | 100GB带宽/月 | ⭐⭐ 一般 | 45min/天 | ✅ |
| Netlify | 100GB带宽/月 | ⭐⭐ 一般 | 300min/月 | ✅ |
| GitHub Pages | 100GB带宽/月 | ⭐ 较差 | 10min/次 | ✅ |

> Cloudflare Pages 在国内有节点，访问速度明显优于其他三个，且无流量限制。

## 四、GitHub 存储风险分析 ⭐

### 你的担心：研报文本多了会不会超限？

**直接结论：纯文本 Markdown 完全不用担心。**

来算一笔账：

| 项目 | 数值 |
|------|------|
| 一篇典型 Markdown 研报大小 | ~30-50 KB |
| 每天生成 5 篇简报 + 1 篇周报 | 约 250 KB/天 |
| 一年研报总大小 | 约 **90 MB** |
| GitHub Pages 站点限制 | **1 GB**（够存 ~11 年） |
| GitHub 仓库 .git 限制 | **10 GB**（推荐上限） |
| 单个文件大小限制 | **100 MB** |

> **即使你每天写 20 篇研报，连写 10 年，也远达不到 GitHub 的限制。Markdown 纯文本的体积在 Git 中还会被进一步压缩（通常压缩比 3:1 ~ 5:1）。**

### 需要注意的

- ⚠️ **图片/图表**：如果研报中包含大量截图或生成的图表 PNG，图片会成为存储大户。建议图表用 SVG（矢量，极小）或托管到 CDN
- ⚠️ **Git LFS**：如果有超过 100MB 的单个文件才需要，你的场景不存在这个问题
- ✅ **.gitignore**：把 `node_modules/`、`.cache/`、构建产物加入忽略

## 八、风险与注意事项

| 风险 | 应对 |
|------|------|
| 数据源接口变动 | 多个 Skill 互为备份（东财/腾讯/同花顺） |
| AI 分析质量不稳定 | 人工复核机制，迭代 Prompt |
| 自动发布出错 | 先推送到 staging，确认后再发布 |
| 合规风险 | 邮件仅发自己，研报标注「仅供参考，不构成投资建议」 |
| Cloudflare 国内偶尔不稳 | 备选 GitHub Pages 镜像 |
